# Design: Anthropic API Support

Adds native Anthropic API as a provider alongside the existing OpenAI-compatible API.
Within Anthropic, supports two auth methods: API key and OAuth (Claude.ai subscription).

---

## 1. API Type System

### Current state

All API configuration lives in `APISettings.js` (`getApiConfig()`) and API connection presets
stored in IndexedDB (`gz_api_connection_presets`). Every preset stores: `endpoint`, `key`, `model`,
`max_tokens`, `context`, `temp`, `topp`, `stream`. There is no concept of API type — everything
assumes OpenAI-compatible format with endpoint + key.

### Changes

Add `apiType` and `authType` fields to API presets:

```js
// API connection preset structure
{
    id: 'my-claude',
    name: 'Claude Max',
    apiType: 'anthropic',       // 'openai' (default) | 'anthropic'
    authType: 'oauth',          // 'key' (default) | 'oauth' (only relevant for anthropic)
    endpoint: '',               // openai only (ignored for anthropic)
    key: '',                    // API key (openai: any key, anthropic: sk-ant-...)
    oauth: null,                // OAuth tokens, per-preset (see section 5)
    model: 'claude-sonnet-4-20250514',
    max_tokens: 8192,
    context: 200000,
    temp: 0.7,
    topp: 0.9,
    stream: true
}
```

OAuth tokens live inside the preset (same as `key`), not in a global storage.
Each preset manages its own token independently. See section 5 for details.

Missing `apiType` field defaults to `'openai'` for backward compatibility.

### getApiConfig() changes

`getApiConfig()` in `APISettings.js` must include `apiType` and `authType` in its return value.
These propagate through `generationService.js` to `llmApi.js` without modification.

## 2. Anthropic Request Format

### Endpoint

All Anthropic requests go to `https://api.anthropic.com/v1/messages` (hardcoded, no user endpoint).

### Request body differences

OpenAI format (current):
```json
{
    "model": "gpt-4",
    "messages": [
        { "role": "system", "content": "You are Alice." },
        { "role": "user", "content": "Hello" },
        { "role": "assistant", "content": "Hi" }
    ],
    "temperature": 0.7,
    "max_tokens": 8000,
    "stream": true
}
```

Anthropic format:
```json
{
    "model": "claude-sonnet-4-20250514",
    "system": "You are Alice.",
    "messages": [
        { "role": "user", "content": "Hello" },
        { "role": "assistant", "content": "Hi" }
    ],
    "temperature": 0.7,
    "max_tokens": 8000,
    "stream": true
}
```

Key differences:
- `system` is a top-level field — supports array of `TextBlockParam` (each block can have
  `cache_control`), which maps naturally from multiple system messages
- No `system` role in messages array
- **Sampling parameters restriction:** `temperature` and `top_p` cannot be used simultaneously
  (breaking change since Claude 4.x). Use one or the other; `temperature` is recommended.
  `top_p` and `top_k` are marked as deprecated (verify: need to find primary source).
  Ref: [Messages API](https://docs.anthropic.com/en/api/messages),
  [Migration guide](https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/token-efficient-tool-use)
- **Thinking** differs significantly from OpenAI:
  - `thinking: { type: 'enabled', budget_tokens: N }` — manual mode (budget_tokens deprecated)
  - `thinking: { type: 'adaptive' }` — recommended, model calibrates reasoning dynamically
  - `display`: `'summarized'` (default on Claude 4, returns summary) or `omitted`. Summarized thinking is generated by a separate model.
  - Streaming: `thinking_delta` events + `signature_delta` before `content_block_stop`
  - Interleaved thinking: auto-enabled with adaptive thinking on Opus/Sonnet 4.6
  Ref: [ThinkingConfigParam](https://platform.claude.com/docs/en/api/messages#thinking_config_param)
  [Extended thinking](https://docs.anthropic.com/en/docs/build-with-claude/extended-thinking)
  Extended thinking is deprecated in Claude 4.6 or newer models (for Haiku?). If using newer models, use adaptive thinking instead.
  [Migrating to Claude Haiku 4.5](https://platform.claude.com/docs/en/about-claude/models/migration-guide#migrating-to-claude-haiku-4-5)
- Image format: `{ type: "image", source: { type: "base64", media_type: "...", data: "..." } }`
  instead of `image_url`

### Conversion strategy

Conversion happens in `llmApi.js` before sending the request. The worker and `generationService.js`
continue building prompts in OpenAI format. `llmApi.js` converts when `apiType === 'anthropic'`:

1. Extract all `system` role messages from `messages[]`
2. Map each to a `TextBlockParam` (`{ type: 'text', text: content }`) in the top-level `system` array
   (preserves block boundaries; allows per-block `cache_control` in future)
3. Remove them from `messages[]`
4. If first remaining message is `assistant`, prepend an empty `user` message (Anthropic requires user-first)
5. Merge consecutive same-role messages (Anthropic forbids adjacent messages with the same role)
6. Convert image format if present
7. Map reasoning config: if reasoning enabled, add `thinking: { type: 'adaptive' }`.
   Remove `include_reasoning` and any OpenRouter/Gemini-specific fields.
   Strip `top_p` if `temperature` is also present (Anthropic forbids both).

### Headers

```js
// API Key auth
{
    'Content-Type': 'application/json',
    'x-api-key': 'sk-ant-...',
    'anthropic-version': '2023-06-01'
}

// OAuth auth
{
    'Content-Type': 'application/json',
    'Authorization': 'Bearer {access_token}',
    'anthropic-version': '2023-06-01',
    'anthropic-beta': 'oauth-2025-04-20'
}
```

No `Authorization: Bearer` for API key auth — Anthropic uses `x-api-key` header instead.

Note: the `anthropic-beta` value for OAuth (`oauth-2025-04-20`) is taken from the claude-code-proxy
source code. Verify it is still current before implementation.

## 3. Anthropic SSE Streaming

### Current OpenAI SSE parsing (llmApi.js)

```
data: {"choices":[{"delta":{"content":"Hello"}}]}
data: {"choices":[{"delta":{"reasoning_content":"thinking..."}}]}
data: [DONE]
```

### Anthropic SSE format

```
event: message_start
data: {"type":"message_start","message":{"id":"...","model":"...","role":"assistant"}}

event: content_block_start
data: {"type":"content_block_start","index":0,"content_block":{"type":"text","text":""}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"Hello"}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"thinking_delta","thinking":"step 1..."}}

event: content_block_stop
data: {"type":"content_block_stop","index":0}

event: message_delta
data: {"type":"message_delta","delta":{"stop_reason":"end_turn"}}

event: message_stop
data: {"type":"message_stop"}
```

### Parsing changes

In the streaming loop of `llmApi.js`, when `apiType === 'anthropic'`:

- Parse `content_block_delta` events:
  - `delta.type === 'text_delta'` → text content (`delta.text`)
  - `delta.type === 'thinking_delta'` → reasoning content (`delta.thinking`)
- Detect end of stream: `message_stop` event (instead of `data: [DONE]`)
- Error handling: `event: error` with `data: {"type":"error","error":{"message":"..."}}`

The extracted text/reasoning feed into the same `onUpdate()` / `onComplete()` callback
pipeline that already exists. No changes needed downstream.

### Non-streaming response

OpenAI:
```json
{ "choices": [{ "message": { "content": "Hello", "reasoning_content": "..." } }] }
```

Anthropic:
```json
{
    "content": [
        { "type": "thinking", "thinking": "..." },
        { "type": "text", "text": "Hello" }
    ],
    "stop_reason": "end_turn"
}
```

Extract text from `content` blocks by type. `thinking` blocks → reasoning, `text` blocks → content.

## 5. Auth: OAuth (Claude.ai Subscription)

### Overview

OAuth 2.0 PKCE flow against Claude.ai. Uses the same `client_id` as Claude Code CLI.
Grants access to `api.anthropic.com/v1/messages` using the user's Claude.ai subscription.

### Fixed parameters

```
client_id:      9d1c250a-e61b-44d9-88ed-5944d1962f5e
authorize_url:  https://claude.ai/oauth/authorize
token_url:      https://console.anthropic.com/v1/oauth/token
redirect_uri:   https://console.anthropic.com/oauth/code/callback
scope:          org:create_api_key user:profile user:inference
```

### Browser context

Authorize URLs are launched in `@capacitor/browser`, which uses **Chrome Custom Tabs** on Android and **SFSafariViewController** on iOS — in-process system browsers that share cookies and user-agent with real Chrome/Safari, so Google treats them as trusted during the Claude.ai login step.

Trade-off: the host app cannot read the URL of pages loaded inside them (hard platform security boundary). To work around this, the authorize URL includes `code=true`, which makes Claude.ai render the authorization code on the callback page itself (same UX as Claude Code CLI). The user copies the code and pastes it back. Both native and web hosts use the same paste-based flow; only the mechanism that opens the authorize URL differs.

### Flow

1. **User taps "Authorize"** in ApiView
2. **Generate PKCE** parameters (synchronous):
   - `code_verifier` = 32 random bytes → base64url
   - `code_challenge` = SHA-256(code_verifier) → base64url (via `js-sha256`,
     not Web Crypto — `crypto.subtle` is unavailable over plain `http://` or
     `file://`, which the Android WebView uses for locally bundled assets)
   - `state` = 32 random bytes → base64url
3. **Build authorization URL**:
   ```
   https://claude.ai/oauth/authorize?
     code=true
     &client_id=9d1c250a-e61b-44d9-88ed-5944d1962f5e
     &response_type=code
     &redirect_uri=https://console.anthropic.com/oauth/code/callback
     &scope=org:create_api_key user:profile user:inference
     &code_challenge={code_challenge}
     &code_challenge_method=S256
     &state={state}
   ```
4. **Open the authorize URL** depending on platform:
   - **Native (Android/iOS):** auto-launch Custom Tabs / SFSafariViewController.
     ```js
     import { Browser } from '@capacitor/browser';
     await Browser.open({ url: authorizationUrl, presentationStyle: 'popover' });
     ```
   - **Web / Electron:** do **not** auto-open (popups trigger the same captcha
     wall as WebViews). Instead, render the authorize URL as a copyable text
     field with a Copy button. The user opens it manually in their own Chrome /
     Firefox / Safari.
5. **User authorizes in the system browser.** After the redirect, the callback
   page at `https://console.anthropic.com/oauth/code/callback` displays the
   authorization code (because of `code=true`). Claude.ai's CLI-style callback
   renders the value as `{code}#{state}`.
6. **User copies the displayed value** and pastes it into the paste field in
   ApiView. On native they dismiss Custom Tabs first (back/X) to return to the
   app; on web they just switch back to the app tab.
7. **Parse the pasted value.** Accepted formats:
   - Full callback URL (`https://.../callback?code=X&state=Y`)
   - Raw `code#state` (Claude.ai's CLI format)
   - Bare code with no state (fallback; state check is skipped — manual paste
     has no CSRF surface)
8. **Validate state** when it was provided. If the paste included a state,
   it must match the PKCE state we generated. If not provided, skip — the user
   is visibly in control.
9. **Exchange code for tokens:**
   ```
   POST https://console.anthropic.com/v1/oauth/token
   Content-Type: application/json

   {
       "grant_type": "authorization_code",
       "code": "{code}",
       "state": "{state}",
       "client_id": "9d1c250a-e61b-44d9-88ed-5944d1962f5e",
       "code_verifier": "{code_verifier}",
       "redirect_uri": "https://console.anthropic.com/oauth/code/callback"
   }
   ```
   Response: `{ access_token, refresh_token, expires_in }`
10. **Save tokens** into the preset's `oauth` field:
    ```js
    preset.oauth = {
        access_token: '...',
        refresh_token: '...',
        expires_at: Date.now() + expires_in * 1000
    };
    saveApiPresets(apiPresets);
    ```
11. **Close the system browser** (native only; no-op on web) and update UI to
    show "Authorized" with expiry time.

### Two-step API

The flow is split into two pure functions — no flow object, no promises, no
state machine:

```js
beginAuthorize() → { pkce, authUrl }
    // Generates a fresh PKCE pair and builds the authorize URL.
    // On native, also opens the system browser. On web, the caller shows
    // authUrl in a copyable field.

completeAuthorize(input, pkce) → Promise<tokens>
    // input: raw code, code#state, or full callback URL.
    // pkce: the object returned by beginAuthorize.
    // Parses the code and POSTs to the token endpoint.
```

The caller keeps `pending = { pkce, authUrl }` as a ref. Tapping Authorize
overwrites it with a fresh attempt; submitting consumes it; logging out
clears it. There is no re-entry guard — a second tap simply regenerates
PKCE and invalidates the first attempt, which is the correct behaviour for
a user who abandoned the first browser window.

There is also no OAuth `state` validation on submit. The `state` parameter
exists to protect redirect-based flows from CSRF, but manual paste is under
direct user control — there is no attacker-controlled redirect to defend
against. The parameter is still generated and sent (the authorize server
requires it) but not checked on the way back.

### Token refresh

Before each API request (in `llmApi.js`), when `authType === 'oauth'`:

1. Load tokens from `preset.oauth`
2. Check `expires_at <= Date.now() + 60000` (1 min buffer)
3. If expired, refresh:
   ```
   POST https://console.anthropic.com/v1/oauth/token
   Content-Type: application/json

   { "grant_type": "refresh_token", "refresh_token": "...", "client_id": "..." }
   ```
4. Save new tokens back into `preset.oauth` (use new `refresh_token` if provided, keep old otherwise)
5. Return valid `access_token`

Only the active preset's token is refreshed. Inactive presets are not touched.

### Race condition protection

If multiple requests from the same preset detect expiry simultaneously, one refresh per preset:

```js
// Map of preset ID -> pending refresh promise
const refreshPromises = new Map();

async function getValidToken(preset) {
    const tokens = preset.oauth;
    if (!tokens) throw new Error('Not authenticated');

    if (tokens.expires_at > Date.now() + 60000) {
        return tokens.access_token;
    }

    if (refreshPromises.has(preset.id)) {
        return refreshPromises.get(preset.id);
    }

    const promise = (async () => {
        try {
            const newTokens = await refreshAccessToken(tokens.refresh_token);
            preset.oauth = {
                access_token: newTokens.access_token,
                refresh_token: newTokens.refresh_token || tokens.refresh_token,
                expires_at: Date.now() + newTokens.expires_in * 1000
            };
            saveApiPresets(apiPresets);
            return preset.oauth.access_token;
        } finally {
            refreshPromises.delete(preset.id);
        }
    })();

    refreshPromises.set(preset.id, promise);
    return promise;
}
```

### Auto-retry on 401

If Anthropic API returns 401:
1. Clear cached token for the active preset
2. Attempt token refresh
3. Retry the request once with new token
4. If still 401, show error "Session expired, please re-authorize"

### Logout

Set `preset.oauth = null`, save presets. Update UI to show "Not authorized" state.

### Historical note: embedded WebView was rejected

An earlier prototype used `@capgo/inappbrowser` (full embedded WebView with `urlChangeEvent`) to capture the callback URL silently. Abandoned because:

1. **Google blocks OAuth in embedded WebViews** ([2016 policy](https://developers.googleblog.com/2016/08/modernizing-oauth-interactions-in-native-apps.html)) — `accounts.google.com` refuses to render the login form in detected embedded user-agents. Since Claude.ai's primary login is Google Sign-In, this broke auth for most users.
2. **Web fallback is a no-op** — `@capgo/inappbrowser`'s web implementation is a stub; `openWebView()` resolves immediately and never emits `urlChangeEvent`. No dev/Electron test path.

The current paste-based flow (`@capacitor/browser` + `code=true`) sidesteps both issues.

## 6. New Module: oauthService.js

New service at `src/core/services/oauthService.js`. Encapsulates all OAuth logic:

```js
// Public exports:
beginAuthorize()                  // Returns { pkce, authUrl }. Opens system browser on native.
completeAuthorize(input, pkce)    // Parses input (code / code#state / full URL),
                                  // exchanges for tokens. Returns Promise<tokens>.
getValidAccessToken(oauth, presetId, onTokenRefresh)  // Returns valid token, refreshing if needed
```

`generatePKCE`, `buildAuthorizationURL`, and the token-exchange/refresh
helpers are kept as module-internal functions — there are no other callers.

### PKCE implementation

Uses `js-sha256` for the digest instead of `crypto.subtle.digest`. Reason:
`crypto.subtle` is only exposed in secure contexts (HTTPS, `localhost`, or
isolated extension contexts). Capacitor on Android bundles assets under
`http://localhost` via a custom scheme handler — in practice `crypto.subtle`
is available there, but it is **not** available when the app is loaded from
`file://` or when built for Electron without a secure loader. Using the
userland `js-sha256` package removes the platform dependency and lets
`generatePKCE()` stay synchronous (it has no `await`).

```js
import { sha256 } from 'js-sha256';

function generatePKCE() {
    const verifierBytes = new Uint8Array(32);
    crypto.getRandomValues(verifierBytes);
    const code_verifier = base64url(verifierBytes);

    const digest = sha256.arrayBuffer(code_verifier);
    const code_challenge = base64url(new Uint8Array(digest));

    const stateBytes = new Uint8Array(32);
    crypto.getRandomValues(stateBytes);
    const state = base64url(stateBytes);

    return { code_verifier, code_challenge, state };
}

function base64url(bytes) {
    return btoa(String.fromCharCode(...bytes))
        .replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
}
```

### Manual-paste flow implementation

The split is entirely in the caller's hands:

- `beginAuthorize()` generates PKCE, builds the URL, and on native calls
  `Browser.open({ url: authUrl })`. It returns immediately — no waiting, no
  listeners, no timeout. On web it just returns the URL and lets the view
  render a copy field.
- `completeAuthorize(input, pkce)` strips `#state` and query params to
  recover the raw code, then POSTs to the token endpoint. It never touches
  the browser — closing the system tab is the user's job (swipe back on
  Custom Tabs, close tab on desktop). This keeps the service stateless: no
  `settled` flag, no `Browser.close`, no timeout, no ownership checks.

### Token storage

Tokens stored inside each API preset's `oauth` field (persisted to IndexedDB via `saveApiPresets()`):
```js
preset.oauth = {
    access_token: '...',
    refresh_token: '...',
    expires_at: 1712345678000
};
```

Tokens are per-preset, same as API keys. Different presets can hold tokens from different
Claude.ai accounts. Only the active preset's token is refreshed on demand.

## 10. Edge Cases

- **Multiple Anthropic presets**: Key and OAuth presets can coexist independently. Each preset
  stores its own `key` or `oauth` tokens. Different OAuth presets can hold tokens from different
  Claude.ai accounts.
- **Token expiry during generation**: Not an issue. Auth is checked at request initiation.
  Once the stream is established, the connection continues regardless of token expiry.
  The token only needs to be valid when starting a new request.
- **Offline**: Token refresh fails. Show error, let user retry when online.
- **Concurrent refreshes**: Protected by per-preset single-promise pattern (section 5).
  Different presets can refresh concurrently without conflict.
- **Migration**: Existing presets without `apiType` default to `'openai'`. No migration needed.
- **Sampling params**: When converting to Anthropic format, if both `temperature` and `top_p`
  are present, drop `top_p` (Anthropic forbids both simultaneously).
